# TASK_NOINT_ - GPIO查询方式控制数码管与LED

## 概述

本程序基于 MicroBlaze 软核处理器，采用**查询方式（Polling）**实现以下功能：

1. **开关（Switch）查询**：通过轮询 GPIO ISR 状态标志寄存器检测拨码开关变化，控制 LED 显示，并通过 UART 打印
2. **按键（Button）查询**：主循环中轮询按键 GPIO 数据寄存器，检测按键按下，将对应字符段码存入显示缓冲区，实现数码管滚动显示
3. **数码管软件扫描**：不使用定时器，直接用软件延时循环扫描 8 位数码管

> 与 `TASK_INT_0`（中断方式）不同，本程序完全通过 **CPU 轮询**实现所有外设的检测和刷新，未使用中断控制器（INTC）和硬件定时器。

## Vivado 硬件设计要点（Block Design 配置）

### 1. AXI GPIO 配置

| 组件 | 配置项 | 必须值 | 说明 |
|------|--------|--------|------|
| **AXI GPIO 0** | GPIO Data Width (Channel 1) | 16 | 16位拨码开关输入 |
| | GPIO Data Width (Channel 2) | 16 | 16位LED输出 |
| | Enable Dual Channel | ✔ 勾选 | 需要两个通道 |
| | Enable Interrupt | ✘ 不勾选 | 本程序不使用中断，仅通过查询 ISR 寄存器检测状态变化 |
| **AXI GPIO 1** | GPIO Data Width (Channel 1) | 8 | 8位数码管位选输出 |
| | GPIO Data Width (Channel 2) | 8 | 8位数码管段码输出 |
| | Enable Dual Channel | ✔ 勾选 | 需要两个通道 |
| | Enable Interrupt | ✘ 不勾选 | 无需中断 |
| **AXI GPIO 2** | GPIO Data Width (Channel 1) | 5 | 5位按键输入，低5位有效 |
| | Enable Dual Channel | ✘ 不勾选 | 仅需通道1 |
| | Enable Interrupt | ✘ 不勾选 | 本程序不使用中断，直接在 main 循环中轮询按键电平 |

> ⚠ **与 TASK_INT_0 的关键区别**：三个 GPIO IP 均**不需要使能中断**，GPIO 0 的中断输出引脚（`ip2intc_irpt`）在硬件上可以悬空不接。程序通过直接读取 `XGPIO_ISR_OFFSET` 寄存器查询开关状态变化（相当于"软件轮询中断标志"）。

> ⚠ **硬件连接注意事项**：
> - GPIO 2 的数据宽度设为 5 位，但 AXI GPIO IP 会向上取整到 8 位，读取时需用 `&0x1f` 屏蔽高 3 位
> - 拨码开关建议硬件上加上拉电阻（或使能 IP 内部的 Pull-Up），确保浮空时读到确定值
> - LED 输出建议加限流电阻（通常 220Ω~1kΩ），数码管段码输出也需匹配对应驱动电路

### 2. AXI Timer 配置

**本程序不需要 AXI Timer IP。** 数码管扫描延时完全由软件循环实现，无需硬件定时器。

因此 Block Design 中可以不例化 AXI Timer，CPU 无需连接定时器外设。

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| Enable Interrupt Controller | ✘ 不必须 | 本程序不使用中断，但 MicroBlaze 内部中断（如异常）可保持默认 |
| Enable AXI Peripherals | ✔ 使能 | 通过 AXI Interconnect 连接 GPIO 0/1/2 |

## 代码结构

### 1. 全局变量与初始化

```c
// GPIO 方向配置
Xil_Out16(GPIO0_BASEADDR + XGPIO_TRI_OFFSET, 0xffff);    // 开关 → 输入
Xil_Out16(GPIO0_BASEADDR + XGPIO_TRI2_OFFSET, 0x0);      // LED  → 输出
Xil_Out8(GPIO1_BASEADDR + XGPIO_TRI_OFFSET, 0x0);        // 位选 → 输出
Xil_Out8(GPIO1_BASEADDR + XGPIO_TRI2_OFFSET, 0x0);       // 段码 → 输出
Xil_Out8(GPIO2_BASEADDR + XGPIO_TRI_OFFSET, 0x1f);       // 按键 → 输入
```

- `segtable[5]`：5 个字符的段码表：`C(0xc6)` `U(0xc1)` `L(0xc7)` `R(0x88)` `d(0xa1)`
- `segcode[8]`：8 位数码管显示缓冲区，初始全部显示 `0xc0`（数字 0）
- `poscode[8]`：8 位数码管位码，**低电平选中**，从左到右依次为 `0x7F` ~ `0xFE`
- `sw`：记录最近一次读到的开关值
- `swFlag`：读取 ISR 查询开关变化标志
- `mask`：记录按下的按键索引（0~4）

### 2. main() 主循环

```c
main()
  ├── 初始化 GPIO 方向
  ├── 打印启动信息
  └── while(1)
        ├── while(有按键按下)      // 等待按键按下
        │     ├── 读取按键值
        │     ├── while(按键未松开) // 忙等待松手
        │     ├── 打印按键编码
        │     ├── 解析按键索引 → mask
        │     ├── 更新显示缓冲区（左移一位，新字符在最右侧）
        │     └── while(无按键按下)  // 持续显示数码管
        │           └── for(i=0→7)   // 循环扫描 8 位数码管
        │                 ├── 输出段码 segcode[i]
        │                 ├── 输出位码 poscode[i]
        │                 └── for(j=0→999) // 延时 + 查询开关
        │                       ├── 读取 ISR 查询开关变化
        │                       └── 若开关变化 → 更新 LED + 打印
        └── (回到外层 while 继续等待按键)
```

## 关键代码分析

### 按键检测逻辑

```c
while(((读取按键) & 0x1f) != 0)    // ① 等待有按键按下（进入循环体）
{
    button = 读取按键 & 0x1f;      // ② 读取按键值
    while((读取按键 & 0x1f) != 0); // ③ 忙等待按键松开（空循环）
    // ... 处理按键 ...
    
    while((读取按键 & 0x1f) == 0)   // ④ 等待所有按键松开后，持续显示数码管
        for(i=0→7)                  // ⑤ 循环扫描 8 位数码管
        {
            输出段码 → 输出位码 → 延时 + 查询开关
        }
}
```

**执行流程：**
1. 步骤①：外层 `while` 等待按键按下
2. 一旦有按键按下，进入循环体
3. 步骤②：读取按键值
4. 步骤③：等待按键松开（阻塞）
5. 处理按键（解析索引、更新缓冲区）
6. 步骤④：进入内层 `while`，等待按键全部松开后开始扫描数码管
7. 步骤⑤：循环扫描 8 位数码管，每次扫描时在延时期检测开关变化

> ⚠ **注意**：步骤①和步骤④的进入条件正好相反：
> - 外层：`(按键值) != 0` → 有按键按下时进入
> - 内层：`(按键值) == 0` → 无按键按下时进入
> 
> 这意味着：当用户按下按键期间，数码管不会扫描（无显示输出）；只有当所有按键松开后，数码管才开始循环显示。

### 开关查询逻辑

开关通过轮询 GPIO ISR 寄存器实现：

```c
swFlag = Xil_In16(XPAR_AXI_GPIO_0_BASEADDR + XGPIO_ISR_OFFSET);  // 读取 ISR
if (swFlag & 0x0001)    // 查询第 0 位是否有状态变化
{
    sw = Xil_In16(XPAR_AXI_GPIO_0_BASEADDR + XGPIO_DATA_OFFSET);  // 读取当前开关值
    Xil_Out16(XPAR_AXI_GPIO_0_BASEADDR + XGPIO_DATA2_OFFSET, sw); // LED 同步显示
    xil_printf("Switch = 0x%X\r\n", sw);
    Xil_Out16(XPAR_AXI_GPIO_0_BASEADDR + XGPIO_ISR_OFFSET, 0x01); // 清除 ISR 标志
}
```

> ℹ GPIO IP 内部有一个 ISR（Interrupt Status Register）寄存器，即使在不使能中断的情况下，该寄存器仍会记录 GPIO 引脚的状态变化（边沿检测）。程序通过轮询这个寄存器来检测开关是否被拨动，这种方式介于"中断"和"直接读取电平"之间。

### 数码管扫描与延时

```c
for (int i = 0; i < 8; i++)   // 循环扫描 8 位数码管
{
    Xil_Out8(..., segcode[i]);   // ① 先输出段码
    Xil_Out8(..., poscode[i]);   // ② 再输出位码（选通当前位）
    
    for (int j = 0; j < 1000; j++)  // ③ 延时 + 查询开关
    {
        // 查询开关变化
        swFlag = Xil_In16(... + XGPIO_ISR_OFFSET);
        if (swFlag & 0x0001) { /* 更新 LED 和打印 */ }
    }
}
```

- 段码输出顺序：**先段码 → 后位码**（见下方"消影区别"章节详细分析）
- 延时循环约 1000 次空操作，其实际时长取决于系统时钟频率
- 在延时期间同时查询开关状态，避免完全阻塞 CPU

## 注意事项

1. **程序逻辑嵌套复杂**：外层 `while` + 内层 `while` + `for` 循环共三层嵌套，代码缩进和括号匹配容易出错，可读性较差。

2. **按键松开前不显示数码管**：当按键被按下时，程序停在步骤③或步骤①中，数码管无输出（全灭）。只有按键完全松开后，数码管才恢复显示。

3. **忙等待阻塞 CPU**：按键松手检测（`while(...!=0)`）是空循环，期间 CPU 无法处理其他任务。如果用户长按按键，开关变化也无法被检测。

4. **开关检测不及时**：开关查询仅在数码管扫描的延时循环中进行，延迟时间取决于 `for(j=0;j<1000;j++)` 的执行时长。如果某位数码管的延时尚未结束，开关变化可能不会被立即检测到。

5. **无消抖处理**：开关和按键均未做硬件或软件消抖，在机械触点抖动期间可能误判多次触发。

6. **查询 ISR 方式**：代码通过读取 GPIO 的 ISR 寄存器来检测开关变化，但并未使能 GPIO 的中断输出。这种方式利用了 GPIO IP 内部的状态变化捕获功能，属于"半查询"方式。

7. **显示缓冲区滚动**：与 TASK_INT_0 相同，每次按键将 segcode 左移一位，新字符插入最右侧。

8. **延时时间不确定**：`for(j=0;j<1000;j++)` 的延时时间取决于编译器优化级别和系统时钟频率，在不同配置下可能差异很大。若频率过高，可能因扫描太快导致亮度不均或闪烁；若频率过低则数码管亮度足够但 CPU 响应变慢。

## 程序查询控制与中断方式动态显示7段数码管时消影的区别

### 两种方式下的段码/位码输出时序对比

| 步骤 | 查询方式（TASK_NOINT_） | 中断方式（TASK_INT_0） |
|------|------------------------|----------------------|
| ① | 输出**段码** segcode[i] | 消影：位选 → 0xFF（全灭） |
| ② | 输出**位码** poscode[i] | 消影：段码 → 0xFF（全灭） |
| ③ | 延时 + 查开关 | 输出**位码** poscode[pos] |
| ④ | (下一轮 i+1) | 输出**段码** segcode[pos] |
| ⑤ | - | 退出 ISR，下次中断 pos++ |

### 为什么中断方式需要消影？

**中断方式的特性：**
- 定时器 ISR 是**异步触发**的，每次进入 ISR 时不知道上一次 GPIO 输出处于什么状态
- 上次中断可能刚设置了某一位的位码，这次中断要切换到下一位
- 如果不先消影，直接输出新的位码和段码，可能发生：
  - 旧位码 + 新段码 → 旧位对应的数码管短暂显示新段码（残影/鬼影）
  - 新位码 + 旧段码 → 新位对应的数码管短暂显示旧段码（残影/鬼影）

**中断方式消影的原理：**
```
① 位选 → 0xFF      // 先关闭所有位（任何数码管都不选通）
② 段码 → 0xFF      // 所有段熄灭（确保无电流残留）
③ 输出新位码       // 选通当前要显示的数码管
④ 输出新段码       // 显示对应的字符
```
此时 GPIO 输出被刷新，不会出现新旧交替的残影。

### 为什么查询方式不需要消影？

**查询方式的特性：**

- 数码管扫描在**主循环的 for 循环中同步执行**，执行顺序完全可控
- 代码输出顺序是：**先段码 → 后位码**
- 当输出段码时，位码还是**上一轮**的值，上一位的数码管会短暂显示新的段码
- 但紧接着立即输出新的位码，将当前位选通，这个过程**极短**（仅几条指令的间隔）

**具体分析：**
```
当前状态：位 i 被选通，显示段码[i]
↓
输出 segcode[i+1]  →  位 i 仍然选通，但段码已变为段码[i+1]
                      （此时位 i 显示错误内容，持续约几十ns）
↓ 立即（下一条指令）
输出 poscode[i+1]  →  切换到位 i+1，显示段码[i+1]
                      （恢复正常）
```

这个"错误显示"的持续时间极短（约 2~3 个总线时钟周期），人眼无法感知，数码管也不会明显增亮，因此**可以省略消影操作**。

### 特别情况：如果查询方式改为"先位码后段码"

```c
// 错误顺序：先选通位，再输出段码
Xil_Out8(..., poscode[i]);   // 选通位 i（此时段码还是上一轮的旧值）
Xil_Out8(..., segcode[i]);   // 输出段码
```

这种情况下：
1. 位 i 被选通 → 显示旧段码（上一轮的段码）
2. 段码更新 → 显示正确的段码

旧段码的残留时间同样只有几十 ns，**实际上人眼也无法感知**，因此即使交换顺序，消影也并非必须。但以下情况**必须消影**：

| 场景 | 必须消影？ | 原因 |
|------|-----------|------|
| 中断方式 | ✅ 必须 | ISR 异步触发，GPIO 状态不可控 |
| 查询方式（先段码后位码） | ❌ 不必须 | 顺序可控，错误显示时间极短 |
| 查询方式（先位码后段码） | ❌ 不必须 | 同上，时间极短 |
| 数码管相距较远/驱动电路有保持电容 | ✅ 建议 | 残留电荷会保持错误显示较长时间 |
| 高刷新率下追求极致显示质量 | ✅ 建议 | 增加消影可消除任何潜在残影 |

### 总结

| 对比项 | 查询方式（Polling） | 中断方式（Timer ISR） |
|--------|-------------------|---------------------|
| 扫描触发方式 | 主循环 for 循环 | 定时器硬件中断 |
| 时序可控性 | ✅ 完全可控（顺序执行） | ❌ 异步触发（状态不可预知） |
| 消影需求 | ❌ 不必须（但加消影无害） | ✅ 必须加消影防止残影 |
| 消影实现 | 输出前先写 0xFF 到 GPIO | 同一操作，在 ISR 开头执行 |
| 消影开销 | 增加 2 条 GPIO 写操作 | 增加 2 条 GPIO 写操作 |
| 推荐做法 | 加消影更规范，防意外 | 必须加消影 |

> 💡 **最佳实践**：无论是查询还是中断方式，都建议在切换数码管显示时加上消影步骤。增加两条 GPIO 写操作的代价极小，但可以避免因硬件差异（如驱动电路有保持电容、GPIO 输出速率慢等）导致的残影问题。
